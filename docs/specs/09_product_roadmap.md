## 1. 現状整理

### 稼働中の機能

| 項目 | 内容 |
|------|------|
| コア機能 | ブログ記事のAI要約 → LINE配信 |
| フレームワーク | Next.js 14 / React 18 / TypeScript |
| データ保存 | ファイルベース（JSONファイル） |
| デプロイ先 | Vercel |
| スタイリング | Tailwind CSS |
| 管理画面 | `/dashboard` に基本UIあり |

### 技術的制約（現状）

- データベース未導入（ファイルベース保存のためスケーラビリティに限界）
- 認証基盤なし
- 外部サービス連携なし（LINE Messaging API以外）
- Supabase等のBaaS未導入

## 2. フェーズ分けロードマップ

### Phase 1: 基盤構築（目安: 4〜6週間）

**目標:** ファイルベースからデータベースへ移行し、全後続機能の土台を整える。

| 成果物 | 説明 |
|--------|------|
| Supabaseプロジェクト構築 | プロジェクト作成、RLS（Row Level Security）設定 |
| DBスキーマ設計・マイグレーション | articles, summaries, line_messages, users テーブル |
| 既存データの移行スクリプト | JSONファイル → Supabase PostgreSQL |
| Supabase Auth導入 | 管理画面のログイン認証 |
| CRUD APIの置き換え | ファイル読み書き → Supabase Client経由 |
| 既存機能の動作確認 | AI要約・LINE配信が移行後も正常動作すること |

**ゲート条件:** 既存のAI要約→LINE配信フローがSupabase上で問題なく稼働すること。

---

### Phase 2: 顧客基盤（目安: 6〜8週間）

**目標:** CRMとユーザープロファイルにより、誰に何を届けるかを制御可能にする。

| 成果物 | 説明 |
|--------|------|
| CRM基本機能 | 顧客一覧、詳細表示、検索・フィルタ |
| セグメント管理 | 条件ベースのセグメント作成（年齢、地域、行動等） |
| タグシステム | 手動タグ・自動タグ付与、タグベースの絞り込み |
| ユーザープロファイル充実 | LINE連携プロファイル、行動履歴、興味関心 |
| セグメント別LINE配信 | 特定セグメントへの記事配信 |

**ゲート条件:** セグメントを作成し、そのセグメントのみにLINE配信が行えること。

---

### Phase 3: 収益化・エンゲージメント（目安: 8〜10週間）

**目標:** ECサイト連携、クーポン配信、予約システムによりビジネス価値を直接生む機能を投入する。

| 成果物 | 説明 |
|--------|------|
| ECサイト連携 | 商品データ取込（Shopify / BASE等）、商品レコメンド |
| クーポン配信 | クーポン生成、セグメント別配信、使用状況追跡 |
| 予約システム | 予約枠管理、Google Meet連携による自動URL発行 |
| LIFF（LINE内Webアプリ） | LINE内でプロファイル編集、クーポン確認、予約操作 |

**ゲート条件:** LIFFアプリ上でクーポン取得と予約が完了できること。

---

### Phase 4: 自動化・分析（目安: 6〜8週間）

**目標:** マーケティングオートメーションと分析ダッシュボードにより、データドリブンな運用を実現する。

| 成果物 | 説明 |
|--------|------|
| マーケティングオートメーション | トリガーベースの自動配信（例: 登録3日後にウェルカムクーポン） |
| シナリオビルダー | ノーコードで配信シナリオを構築するUI |
| 分析ダッシュボード | 配信実績、開封率、クーポン利用率、予約率の可視化 |
| レポート自動生成 | 週次・月次レポートの自動作成 |

**ゲート条件:** 自動配信シナリオが1本以上本番稼働し、ダッシュボードで効果を確認できること。

## 3. 優先順位マトリクス（Impact vs Effort）

```
Impact（ビジネスインパクト）
  高 │  ★ EC連携+         ★ マーケティング
     │    クーポン配信       オートメーション
     │
     │  ★ CRM機能         ★ 分析
     │                      ダッシュボード
     │
  中 │  ★ Supabase        ★ 予約システム
     │    DB移行             (Google Meet)
     │
     │  ★ ユーザー        ★ LIFF
     │    プロファイル
  低 │
     └──────────────────────────────────
        低                中              高
                    Effort（実装コスト）
```

| 機能 | Impact | Effort | 優先度 | 判定 |
|------|--------|--------|--------|------|
| Supabase DB移行 | 中 | 低〜中 | **最優先** | 全機能の前提条件。最初に着手必須 |
| CRM機能 | 高 | 中 | **高** | セグメント配信の基盤、収益化への入口 |
| ユーザープロファイル | 中 | 低 | **高** | CRMと同時に構築可能、低コストで効果大 |
| EC連携+クーポン | 高 | 中 | **高** | 直接的な収益貢献 |
| 予約システム | 中 | 中 | **中** | ビジネスモデル次第で優先度変動 |
| LIFF | 中 | 中〜高 | **中** | UX向上だがLINE固有の学習コストあり |
| MAツール | 高 | 高 | **中** | 効果は大きいがCRM・セグメント完成が前提 |
| 分析ダッシュボード | 高 | 中〜高 | **中** | データ蓄積後に最大効果を発揮 |

## 4. MVP定義

### 4.1 Supabase DB移行 MVP

| 項目 | MVP範囲 | MVP外（将来対応） |
|------|---------|-------------------|
| テーブル | articles, summaries, line_messages, users | audit_logs, analytics_events |
| 認証 | メール+パスワードによる管理画面ログイン | OAuth連携、多要素認証 |
| RLS | 基本的な行レベルセキュリティ | 細粒度のロール別ポリシー |
| マイグレーション | 手動実行スクリプト | CI/CDパイプライン統合 |

### 4.2 CRM機能 MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| 顧客管理 | 一覧表示、検索、詳細ページ | 一括インポート/エクスポート |
| セグメント | 手動条件設定（最大3条件のAND） | 動的セグメント、複雑な条件式 |
| タグ | 手動タグ付け、タグ一覧管理 | 自動タグ付け、機械学習ベースのタグ提案 |
| 配信連携 | セグメント指定でLINE配信 | A/Bテスト、配信スケジュール最適化 |

### 4.3 ユーザープロファイル MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| 基本情報 | 名前、LINE表示名、メモ | カスタムフィールド |
| 行動履歴 | 記事閲覧、配信受信の記録 | スコアリング、行動予測 |
| 表示 | プロファイル詳細ページ | タイムライン表示、行動グラフ |

### 4.4 EC連携+クーポン配信 MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| EC連携 | 1プラットフォーム（Shopify or BASE）の商品取込 | マルチプラットフォーム、在庫同期 |
| クーポン | 固定割引クーポン生成、手動配信 | 動的割引、パーソナライズクーポン |
| 追跡 | クーポン発行数・使用数の集計 | コンバージョンファネル分析 |

### 4.5 予約システム MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| 予約枠 | 管理者が手動で日時枠を設定 | カレンダー同期、空き枠自動算出 |
| Google Meet | 予約確定時にMeet URLを自動生成 | Zoom/Teams連携 |
| 通知 | 予約確定のLINE通知 | リマインダー、キャンセル待ち |

### 4.6 LIFF MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| 機能 | プロファイル閲覧、クーポン一覧表示 | フルEC体験、予約管理 |
| UI | モバイル最適化のシンプルな画面 | リッチアニメーション、PWA対応 |
| 認証 | LIFFのgetProfile()によるLINEユーザー連携 | 外部アカウント連携 |

### 4.7 マーケティングオートメーション MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| トリガー | 友だち追加、タグ付与の2種類 | Webhook、外部イベント連携 |
| アクション | LINE配信、タグ付与の2種類 | 分岐条件、待機ステップ |
| シナリオ | 直線的な3ステップまで | 分岐付きシナリオ、A/Bテスト |

### 4.8 分析ダッシュボード MVP

| 項目 | MVP範囲 | MVP外 |
|------|---------|-------|
| 指標 | 配信数、フォロワー数推移、クーポン利用率 | LTV分析、コホート分析 |
| 期間 | 日次・週次・月次の切替 | カスタム期間、リアルタイム |
| 表示 | 基本的な折れ線・棒グラフ | インタラクティブドリルダウン |

## 6. リスク管理

### 6.1 技術リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| DB移行時のデータ損失 | 高 | 低 | 移行前に全JSONファイルのバックアップ取得。段階的移行（読み取りは旧系、書き込みは新旧両方）で切り替え |
| Supabase無料枠の超過 | 中 | 中 | 初期はProプラン（月$25）で開始。使用量モニタリングを設定し、閾値到達時にアラート |
| LINE API Rate Limit | 中 | 中 | セグメント配信時のバッチ処理実装。キューイング（Supabase Edge Functions + pg_cron）で分散送信 |
| LIFF SDKのバージョン互換 | 低 | 中 | LIFF SDK v2を使用し、公式ドキュメントに準拠。LINE Developersの更新情報を定期チェック |
| Google Calendar/Meet API認証の複雑さ | 中 | 中 | サービスアカウント方式を採用。OAuth同意画面の審査を早期に申請 |
| Vercelのサーバーレス関数タイムアウト | 中 | 中 | 重い処理はSupabase Edge FunctionsまたはBackground Functionsに分離 |

### 6.2 ビジネスリスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| 機能過多によるリリース遅延 | 高 | 高 | 各Phase内でもMVP→改善のサイクルを徹底。スコープクリープを防ぐため、Phase完了条件を厳格に運用 |
| ユーザーがCRM/MA機能を使いこなせない | 中 | 中 | シンプルなUIを最優先。テンプレートやウィザード形式で初期設定をガイド |
| EC連携先プラットフォームのAPI変更 | 中 | 低 | アダプターパターンで連携層を抽象化。特定プラットフォームへの密結合を回避 |
| LINE公式アカウントのプラン変更 | 高 | 低 | 配信数の最適化（不要な配信の削減）。メッセージ数を常にモニタリング |
| 個人情報保護法への準拠 | 高 | 中 | プライバシーポリシーの整備。Supabase RLSによるデータアクセス制御。データ削除機能の実装 |

### 6.3 リスク対応の優先順位

1. **即時対応:** DB移行時のデータ損失防止 → バックアップ手順の確立
2. **早期対応:** 個人情報保護法準拠 → プライバシーポリシー策定
3. **計画的対応:** スコープクリープ防止 → Phase完了条件の明文化
4. **監視継続:** API Rate Limit、Supabase使用量 → モニタリング設定
