# 05 ダッシュボード UI/UX 仕様書

LineMag 管理ダッシュボードの UI/UX 設計仕様。既存の4画面（配信ウィザード、スケジュール、配信履歴、ログ）に加え、CRM管理・クーポン管理・予約管理・分析ダッシュボードの4領域を追加する。

---

## 1. 情報アーキテクチャ

### 1.1 サイトマップ

```
/dashboard
├── /                        ... 配信ウィザード（既存）
├── /schedule                ... スケジュール配信（既存）
├── /history                 ... 配信履歴（既存）
├── /logs                    ... 実行ログ（既存）
│
├── /crm                     ... CRM管理
│   ├── /                    ... 顧客一覧
│   ├── /[customerId]        ... 顧客詳細
│   └── /segments            ... セグメント管理
│       ├── /                ... セグメント一覧
│       └── /[segmentId]     ... セグメント編集
│
├── /coupons                 ... クーポン管理
│   ├── /                    ... クーポン一覧
│   ├── /new                 ... クーポン作成
│   ├── /[couponId]          ... クーポン詳細・編集
│   └── /[couponId]/stats    ... 利用状況
│
├── /reservations            ... 予約管理
│   ├── /                    ... 予約一覧（リスト表示）
│   ├── /calendar            ... カレンダー表示
│   └── /slots               ... スロット設定
│       ├── /                ... スロット一覧
│       └── /[slotId]        ... スロット編集
│
└── /analytics               ... 分析ダッシュボード
    ├── /                    ... KPI サマリー
    └── /reports             ... 詳細レポート・グラフ
```

### 1.2 ナビゲーション構造

既存のトップバー水平タブをサイドバーナビゲーションに移行し、機能グループごとにセクション分けする。

```
[サイドバー]
──────────────────
LineMag（ロゴ）

配信管理
  マニュアル配信      /dashboard
  スケジュール        /dashboard/schedule
  配信履歴            /dashboard/history
  ログ                /dashboard/logs

顧客管理
  顧客一覧            /dashboard/crm
  セグメント          /dashboard/crm/segments

クーポン
  クーポン一覧        /dashboard/coupons
  新規作成            /dashboard/coupons/new

予約
  予約一覧            /dashboard/reservations
  カレンダー          /dashboard/reservations/calendar
  スロット設定        /dashboard/reservations/slots

分析
  KPI サマリー        /dashboard/analytics
  詳細レポート        /dashboard/analytics/reports
──────────────────
```

### 1.3 ナビゲーション設計原則

| 原則 | 説明 |
|------|------|
| 深さ制限 | URLパスは最大3階層（`/dashboard/{section}/{subsection}`）に抑える |
| セクションラベル | 各グループに「配信管理」「顧客管理」等の見出しを配置し、視覚的に分類 |
| アクティブ表示 | 現在のパスに一致するアイテムに `bg-green-100 text-green-700` を適用（既存パターン踏襲） |
| パンくず | 3階層目以降のページではヘッダー直下にパンくずリストを表示 |

## 2. ダッシュボードのサイドバー構成

### 2.1 レイアウト変更：トップバー → サイドバー

既存の水平タブ型ナビゲーション（`layout.tsx` の `<nav>` 内 `NAV` 配列）を、左サイドバー+トップバーの組み合わせに移行する。

```
┌──────────────────────────────────────────────────────┐
│ [トップバー] ロゴ                    通知  ユーザー  │
├──────────┬───────────────────────────────────────────┤
│          │                                           │
│ サイド   │  メインコンテンツ                          │
│ バー     │  （max-w-5xl は廃止、                      │
│ 240px    │   サイドバー右の残り幅いっぱいに展開）      │
│          │                                           │
│          │                                           │
│          │                                           │
└──────────┴───────────────────────────────────────────┘
```

### 2.2 サイドバーコンポーネント仕様

**ファイル**: `src/components/dashboard/Sidebar.tsx`

```
Props: なし（パスから状態を導出）
状態: collapsed: boolean（モバイル時のドロワー開閉）
```

| 要素 | 仕様 |
|------|------|
| 幅 | デスクトップ: `w-60`（240px）、折りたたみ時: `w-16`（64px、アイコンのみ） |
| 背景 | `bg-white border-r border-slate-200` |
| セクション見出し | `text-[10px] font-bold text-slate-400 uppercase tracking-widest px-4 pt-4 pb-1` |
| ナビアイテム | `px-4 py-2 text-sm font-medium rounded-lg mx-2` |
| アクティブ | `bg-green-50 text-green-700 font-semibold` |
| ホバー | `hover:bg-slate-50 text-slate-600` |
| アイコン | 各項目の左に 20x20 SVG アイコン、アイコンとラベルの間に `gap-3` |

### 2.3 トップバーコンポーネント仕様

**ファイル**: `src/components/dashboard/TopBar.tsx`

| 要素 | 仕様 |
|------|------|
| 高さ | `h-14`（既存と同一） |
| 左端 | モバイル時のハンバーガーメニューボタン |

| 右端 | 通知ベルアイコン + ユーザーアバター/ドロップダウン |

## 3. 各画面のワイヤーフレーム仕様

### 3.1 既存画面（変更なし）

既存4画面のコンテンツ部分はそのまま維持する。ナビゲーション部分のみサイドバーに移行。

- `/dashboard` — 配信ウィザード（fetch → select → template → confirm の4ステップ）
- `/dashboard/schedule` — スケジュール配信設定（トグル、時刻、テンプレート選択）
- `/dashboard/history` — テーブル形式の配信履歴
- `/dashboard/logs` — フィルター付きログテーブル

**主要コンポーネント**:

| コンポーネント | 説明 |
|--------------|------|
| `CustomerTable` | ソート可能テーブル。列: アバター、名前、LINE表示名、セグメント（バッジ）、登録日、最終来店日 |
| `FilterBar` | ドロップダウンフィルター群。セグメント、登録日範囲、最終来店日範囲、タグ |

| `Pagination` | ページネーション。1ページ20件、`limit` / `offset` パラメータ |

1. 一覧表示 → フィルター/検索で絞り込み
2. 行クリック → `/dashboard/crm/[customerId]` に遷移
3. CSVエクスポートボタン → バックグラウンドでCSV生成、ダウンロード

## 4. レスポンシブ対応方針

### 4.1 ブレークポイント定義

Tailwind CSS のデフォルトブレークポイントを使用する。

| ブレークポイント | 幅 | 分類 |
|----------------|------|------|
| デフォルト | 0-639px | モバイル |
| `sm` | 640px- | モバイル（横向き） |
| `md` | 768px- | タブレット |
| `lg` | 1024px- | デスクトップ |
| `xl` | 1280px- | ワイドデスクトップ |

### 4.2 レイアウト切り替え

| 画面要素 | モバイル (< 768px) | タブレット (768px-1023px) | デスクトップ (1024px+) |
|---------|-------------------|-------------------------|----------------------|
| サイドバー | 非表示（ハンバーガーメニューでドロワー表示） | 折りたたみ表示（アイコンのみ `w-16`） | 展開表示（`w-60`） |
| メインコンテンツ | `p-4`、全幅 | `p-5`、全幅 | `p-6`、全幅 |
| テーブル | 横スクロール or カード形式に変換 | 横スクロール | 通常テーブル |
| 2カラムレイアウト | 1カラムスタック | 1カラムスタック | 2カラム並列 |
| カード型グリッド | 1列 | 2列 | 3列 |
| KPIカード | 2列 x 2行 | 4列 x 1行 | 4列 x 1行 |
| カレンダー | 日表示のみ | 週表示 | 月表示 |

### 4.3 モバイル固有の対応

| 対応項目 | 詳細 |
|---------|------|
| テーブルのカード変換 | `md` 未満では `<table>` を非表示にし、カードリスト形式で同一データを表示する。CSS のみで切り替え（`hidden md:table` / `md:hidden`） |
| タッチターゲット | ボタン、リンクの最小タッチ領域を 44x44px 以上に確保 |
| ドロワーナビ | サイドバーを `fixed inset-y-0 left-0 z-50` のオーバーレイで表示。背景に半透明オーバーレイ（`bg-black/50`）を配置し、タップで閉じる |
| ボトムシート | モーダルはモバイルではボトムシート形式に変換（`fixed bottom-0 inset-x-0 rounded-t-2xl`） |

### 4.4 既存スタイルとの互換性

既存画面で使用されている以下のスタイルパターンをそのまま踏襲する。

| パターン | クラス例 |
|---------|---------|
| カード | `bg-white rounded-xl border border-slate-200 overflow-hidden` |
| セクションヘッダー | `px-5 py-4 border-b border-slate-100` |
| バッジ（成功） | `bg-green-100 text-green-700 rounded-full px-2 py-0.5 text-xs font-medium` |
| バッジ（失敗） | `bg-red-100 text-red-700 rounded-full px-2 py-0.5 text-xs font-medium` |
| 入力フィールド | `px-3 py-1.5 text-sm border border-slate-200 rounded-lg focus:outline-none focus:ring-2 focus:ring-green-500/20 focus:border-green-400` |
| プライマリボタン | `bg-green-600 text-white text-sm font-medium rounded-lg hover:bg-green-700 disabled:opacity-50 transition-colors` |
| セカンダリボタン | `text-slate-600 bg-white border border-slate-200 rounded-lg hover:bg-slate-50 text-sm font-medium` |
| ページタイトル | `text-xl font-bold text-slate-800` |
| サブテキスト | `text-sm text-slate-400 mt-1` |

## 5. アクセシビリティ要件

### 5.1 WCAG 2.1 AA 準拠目標

| カテゴリ | 要件 | 実装方法 |
|---------|------|---------|
| 知覚可能 | コントラスト比 4.5:1 以上（通常テキスト）、3:1 以上（大テキスト） | Tailwind のスレートカラーパレット（`slate-800` on `white` = 約 12.6:1）は既に適合。カスタムカラー追加時はコントラストチェッカーで検証 |
| 知覚可能 | 色のみに依存しない情報伝達 | ステータスバッジにはテキストラベルを必ず併記（既存パターンで実施済み：「成功」「失敗」のテキスト表示） |
| 操作可能 | キーボードナビゲーション | すべてのインタラクティブ要素に `tabIndex` を設定。サイドバーは `Tab` / `Shift+Tab` で移動、`Enter` で選択 |
| 操作可能 | フォーカスインジケーター | `focus:ring-2 focus:ring-green-500/20 focus:border-green-400` を全フォーム要素に適用（既存パターン踏襲） |
| 操作可能 | スキップリンク | `<main>` の前に「メインコンテンツへスキップ」リンクを配置（`sr-only focus:not-sr-only`） |
| 理解可能 | フォームラベル | すべての入力フィールドに `<label>` を紐付け。`htmlFor` と `id` のペアを必ず設定 |
| 理解可能 | エラーメッセージ | フォームバリデーションエラーは該当フィールド直下に赤文字で表示、`aria-describedby` で紐付け |
| 堅牢 | セマンティック HTML | `<nav>`, `<main>`, `<aside>`, `<header>`, `<section>`, `<table>` の適切な使用 |

### 5.2 ARIA 属性ガイドライン

| 要素 | 必須 ARIA 属性 |
|------|---------------|
| サイドバー | `<aside role="navigation" aria-label="メインナビゲーション">` |
| モバイルドロワー | `aria-expanded`, `aria-controls` をハンバーガーボタンに設定 |
| タブ | `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-selected`, `aria-controls` |
| モーダル/ダイアログ | `role="dialog"`, `aria-modal="true"`, `aria-labelledby` |
| ステータスバッジ | `role="status"`（動的更新時） |
| テーブル | `<th scope="col">`, ソート可能列に `aria-sort` |
| ページネーション | `<nav aria-label="ページネーション">`, 現在ページに `aria-current="page"` |
| ローディング | `aria-busy="true"` をコンテンツ領域に設定、`role="status"` でスクリーンリーダーに通知 |

## 6. 状態管理・画面遷移フロー

### 6.1 状態管理方針

| レイヤー | 管理方法 | 対象データ |
|---------|---------|-----------|
| サーバー状態 | `fetch` + `useState`（既存パターン踏襲） | API レスポンスデータ（顧客一覧、予約一覧、クーポン一覧等） |
| URL状態 | Next.js `useSearchParams` + `useRouter` | フィルター条件、ページ番号、ソート順、タブ選択 |
| ローカルUI状態 | `useState` | モーダル開閉、フォーム入力値、ローディングフラグ |
| サイドバー状態 | `useState` in `layout.tsx` | ドロワー開閉（モバイル）、折りたたみ状態（タブレット） |

**方針**: 外部状態管理ライブラリ（Redux, Zustand 等）は現時点では導入しない。既存コードベースの `useState` + `fetch` パターンを踏襲し、複雑化した場合に段階的に導入を検討する。

### 6.2 データフェッチパターン

既存の `useEffect` + `fetch` パターンを標準とする。

```
[ページロード] → useEffect → fetch(API) → useState に格納 → レンダリング
[フィルター変更] → useSearchParams 更新 → useEffect 再実行 → 再フェッチ
[作成/更新/削除] → fetch(API, POST/PUT/DELETE) → 成功時: データ再フェッチ or ローカル更新
```

### 6.3 主要画面遷移フロー

#### CRM フロー

```
顧客一覧 ─[行クリック]──→ 顧客詳細 ─[編集ボタン]──→ 編集モーダル（インライン）
    │                        │
    │                        ├─[クーポン履歴タブ]──→ クーポン詳細（遷移）
    │                        ├─[予約履歴タブ]──→ 予約詳細（遷移）
    │                        └─[戻る]──→ 顧客一覧
    │
    └─[セグメント]──→ セグメント一覧 ─[編集]──→ セグメント編集モーダル
```

#### クーポンフロー

```
クーポン一覧 ─[新規作成]──→ クーポン作成 ─[保存]──→ クーポン一覧
    │
    ├─[カードクリック]──→ クーポン詳細 ─[利用状況]──→ 利用状況ページ
    │                        │
    │                        ├─[編集]──→ 編集フォーム（同一ページ内）
    │                        └─[削除]──→ 確認ダイアログ → 一覧に戻る
    │
    └─[ステータスタブ切替]──→ 一覧フィルタリング（URL パラメータで管理）
```

#### 予約フロー

```

    │
    ├─[カレンダー切替]──→ カレンダー表示 ─[日付クリック]──→ 日別詳細展開
    │                                      │
    │                                      └─[予約クリック]──→ 予約詳細モーダル
    │
    └─[スロット設定]──→ スロット設定 ─[保存]──→ 設定反映
```

### 6.4 ローディング・エラー状態

既存パターンに準拠した統一的な状態表示を行う。

| 状態 | 表示 | 実装 |
|------|------|------|
| 初期ローディング | テーブル/カードエリアにスピナー + 「読み込み中...」テキスト | 既存の `Spinner` SVG コンポーネントを共通化して再利用 |
| データなし | 空状態メッセージ + アクション誘導 | `EmptyState` 共通コンポーネント（アイコン + メッセージ + CTAボタン） |
| エラー | 赤バナー + リトライボタン | `bg-red-50 text-red-700 border border-red-200`（既存パターン） |
| 保存中 | ボタンを `disabled` + テキスト変更（「保存中...」） | 既存の `saving` state パターン踏襲 |
| 成功通知 | 緑バナー（3秒後に自動消去） | `bg-green-50 text-green-700 border border-green-200`（既存パターン） |
| オプティミスティック更新 | 即座にUI更新、失敗時にロールバック | ステータス変更（予約確認等）の即応性向上に適用 |
